## CI/CD 工作流配置

文件路径：`.github/workflows/deploy.yml`

```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches: [main, master]
  workflow_dispatch:       # 支持手动触发

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: true

jobs:
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/configure-pages@v5
      - uses: actions/upload-pages-artifact@v3
        with:
          path: '.'
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

### 工作流说明

| 步骤 | Action | 作用 |
|------|--------|------|
| 1 | `actions/checkout@v4` | 拉取仓库代码 |
| 2 | `actions/configure-pages@v5` | 配置 Pages 环境 |
| 3 | `actions/upload-pages-artifact@v3` | 只上传 `index.html` 作为部署产物（避免把 `uploads/` 原图一起打包） |
| 4 | `actions/deploy-pages@v4` | 将产物部署到 GitHub Pages |

> 触发方式：push 到 `main` **或** `master` 自动触发，也可在 Actions 页面手动 `workflow_dispatch`。

---

## GitHub Pages 设置

仓库 **Settings → Pages** 中需配置：

- **Source**：选择 `GitHub Actions`（不是 Deploy from a branch）
- 选中后，部署完全由 `.github/workflows/deploy.yml` 驱动

## 日常更新流程（修正版）

> 关键点：网页的**唯一生成源**是 `gen_map4.py` + `uploads/`，`index.html` 是脚本生成的产物，**不要手写修改 index.html**。
> `gen_map4.py` 已改为相对路径，输出直接写到仓库根目录 `index.html`，无需再手动复制。

### 改内容的三种场景

| 想改什么 | 怎么改 | 重新生成 |
|----------|--------|----------|
| 某栋楼的描述/坐标/导航点 | 编辑 `gen_map4.py` 里 `BUILDINGS` 中对应字典的 `desc` / `x` / `y` / `nav` | 重跑脚本 |
| 新增/替换某栋楼的照片 | 把图片放进 `uploads/`（注意文件名），在 `BUILDINGS` 的 `photos` 列表里引用 | 重跑脚本 |
| 新增一栋楼 | 往 `BUILDINGS` 追加一个字典（`x`/`y` 是百分比坐标，0–100） | 重跑脚本 |

### 重新生成（本地）

```bash
# 需要 Pillow：pip install pillow
python gen_map4.py
# 输出：仓库根目录 index.html（约 1.2MB，自包含）
```

### 部署上线（固定流程，背这一条就行）

> ⚠️ **给小白：master 和 main 有啥区别？**
> 它们本质是**同一个地位的东西——都是"主分支"**，只是名字不同（GitHub 新仓库默认叫 `main`，老习惯叫 `master`）。
> 本项目之前本地是 `master`、线上是 `main`，两边历史没对上，所以直接 push 会冲突。
> 我已经把它们**统一到同一个提交**，并且把部署工作流改成**两个分支都会触发部署**。
> **结论：你以后只管一直用 `master` 推送即可**（下面命令全写 `master`），别两个分支换来换去切，就不会乱。

每次改完内容，按下面 4 步走：

```bash
# ① 改内容：编辑 gen_map4.py（描述/坐标/楼栋） 或 往 uploads/ 放新图并在 photos 引用
# ② 重新生成网页（需要 Pillow：pip install pillow）
python gen_map4.py
#     → 输出仓库根目录 index.html（图片已转 base64 内嵌，自包含）

# ③ 提交（用 git devlog 结构化提交，禁止裸 git commit）
git devlog -f index.html .github/workflows/deploy.yml gen_map4.py \
  -t feat -m "更新地图内容说明" \
  -v "本地浏览器打开 index.html 预览正常"

# ④ 推送上线（绕过代理，触发 GitHub Actions 自动部署）
git -c http.proxy="" -c https.proxy="" push origin master
```

推送后约 1-2 分钟完成部署，可在以下页面查看进度：

- Actions 日志：https://github.com/wWSsaii/hainnu-map/actions
- 部署完成后访问：https://wWSsaii.github.io/hainnu-map/

### 不想上传"不必要的文件"怎么办（.gitignore）

> 核心原理：`index.html` 是**自包含**的——所有图片都已转成 base64 嵌进 HTML 里了。
> 所以线上部署**只需要 `index.html` 一个文件**，原始 `uploads/` 图片、本地记忆 `.workbuddy/` 都不用上 GitHub。

仓库根已加了 `.gitignore`，会自动忽略以下东西，你**就算手滑 `git add .` 也不会传上去**：

```
uploads/        # 26 张原始图片，index.html 已内嵌，不必入库（避免仓库膨胀）
.workbuddy/     # 本地工作区记忆，纯本地用
outputs/        # 本地生成缓存目录
```

保险起见，提交时**只指定要提交的文件**（像上面第 ③ 步用 `git devlog -f index.html ...`），比 `git add .` 更可控。

> 若哪天换了电脑要重新生成网页，本地 `uploads/` 原图需要自己保留备份（它不在 GitHub 上）。

---

> **凭证 / 代理提示**：若 `git push` 报 SSL 握手失败（之前配过 `http.proxy`），用
> `git -c http.proxy="" -c https.proxy="" push origin master` 临时绕过代理。
> 若报无法读取用户名，先 `git config --global credential.helper manager` 走 Windows 凭据管理器。

## 已知问题与排查

### 1. Git 代理导致 HTTPS 连接失败

**现象**：`git push` 报 `schannel: failed to receive handshake, SSL/TLS connection failed`

**原因**：全局 git 配置了 `http.proxy = http://127.0.0.1:7897`，代理无法正确转发 GitHub HTTPS 请求。

**解决**：推送时临时绕过代理

```bash
git -c http.proxy="" -c https.proxy="" push origin main
```

### 2. 非交互环境凭证问题

**现象**：推送时报 `could not read Username for 'https://github.com'`

**原因**：`credential.helper` 未配置或终端无法弹出认证窗口。

**解决**：

```bash
git config --global credential.helper manager
```

正常情况下会调用 Windows Credential Manager 弹窗完成认证，之后凭证会缓存。

### 3. Pages 返回 404

**排查清单**：

- [ ] Settings → Pages → Source 是否选了 `GitHub Actions`
- [ ] Actions 中最近一次 Run 是否全绿（conclusion = success）
- [ ] 仓库根目录是否有 `index.html`
- [ ] URL 用户名大小写是否正确（`wWSsaii` 而非 `wssaii`）

### 4. Actions 中出现红色失败

进入 https://github.com/wWSsaii/hainnu-map/actions 点击对应 Run，展开标红步骤查看日志。常见原因：

- `index.html` 不在仓库根目录
- `deploy.yml` YAML 语法错误
- Pages Source 未设为 `GitHub Actions`

## 备选部署方案

如未来需要更灵活的托管（自定义域名、CDN 加速等），可考虑：

| 方案 | 优点 | 适用场景 |
|------|------|----------|
| GitHub Pages（当前） | 免费、零配置、自动部署 | 个人项目、静态站 |
| Vercel | 免费 CDN、可绑域名 | 需要全球加速 |
| Netlify | 免费 CDN、表单收集 | 需要表单/后端能力 |
| CloudStudio | 国内访问快 | 面向国内用户 |

切换时只需将 `index.html` 部署到对应平台，无需改动源码。
